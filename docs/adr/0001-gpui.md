# ADR-0001: UI フレームワークとして GPUI を採用する

- Status: Accepted
- Date: 2026-06-12

## Context

コミットグラフ中心の Git GUI クライアントを Rust で作る。グラフ描画(数千行の仮想化リスト + lane のカスタム描画)を 60fps で行いたい。候補: GPUI / egui / iced / Tauri(web view)。

## Decision

GPUI(Zed の UI フレームワーク)を採用する。

## Rationale

- GPU レンダリング前提で、大量行の仮想化リスト(`uniform_list`)とカスタム描画(canvas/paint API)が実績付き(Zed 本体の editor / git panel)。コミットグラフの要件に合う。
- Rust ネイティブで web view を挟まない。Tauri 案は graph 描画を結局 JS で書くことになり「Rust + GPUI」というプロジェクト前提に反する。
- egui は immediate mode で手軽だが、テキスト品質・複雑なレイアウト・ネイティブ感で GPUI に劣る。iced はリスト仮想化とエコシステムが弱い。
- Zed 本体という大規模リファレンス実装があり、API 利用例に困らない。

## 依存指定(2026-06-12 調査で確定)

- gpui は crates.io に公開済み。最新は **0.2.2**(2025-10-22)。`gpui = "0.2.2"` で固定する。
- git main は `gpui` + `gpui_platform` にクレート分割済みで API が変わっている(`Application::new()` → `gpui_platform::application()`)。`gpui_platform` は crates.io 未公開のため、**main 追従はせず crates.io 0.2.2 系を使う**。更新は意図的にのみ行う。
- 一次資料: `docs.rs/gpui`(0.2.2)、docs.rs 上の同梱 examples(`hello_world.rs`)、Zed リポジトリ。gpui.rs はランディングページでありドキュメントサイトではない。

## ビルド環境制約(本開発機で確認済み)

- gpui の build.rs は Metal シェーダを `xcrun metal` でビルド時コンパイルするため通常は **full Xcode が必要**。本開発機は CLT のみで `metal` が使えないことを確認済み。
- → **`runtime_shaders` feature を有効にする**(シェーダを実行時コンパイルし、ビルド時の Metal toolchain 依存を回避)。`gpui = { version = "0.2.2", features = ["runtime_shaders"] }`

## Consequences / Risks

- GPUI は pre-1.0 で破壊的変更が頻繁(README にも明記)。バージョン固定 + Cargo.lock で防御する。
- ドキュメントが薄く、docs.rs と Zed のソース(crates/gpui/examples)を一次資料として読む必要がある。
- macOS が最も成熟。Linux/Windows 対応は best-effort とする(requirements の非機能要件に反映済み)。
