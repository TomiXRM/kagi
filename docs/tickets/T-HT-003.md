# T-HT-003: Pull(fetch + merge)の plan + 実行

- Status: done
- 担当: Sonnet subagent
- 依存: T-HT-001(ADR-0009 / ADR-0005)

## 背景

Toolbar の Pull を実装する。MVP は **merge pull のみ**(rebase は later)。
方針(ADR-0009): ネットワーク(fetch)だけ CLI、**merge 自体は libgit2 の in-memory 方式**で
「conflict するなら実行せず repo 無傷」を保証する(cherry-pick T016 と同じ哲学)。

## スコープ

### 1. CLI wrapper(`src/git/cli.rs` 新規)

```rust
pub struct GitCliOutput { pub status: i32, pub stdout: String, pub stderr: String }
pub fn run_git(repo: &Path, args: &[&str]) -> Result<GitCliOutput, GitError>
```
- `std::process::Command`(shell 経由禁止・引数配列)
- env: `GIT_TERMINAL_PROMPT=0`(認証プロンプトでハングさせない)、`LC_ALL=C`
- 60秒 timeout(`wait_timeout` は std にないので、spawn + ポーリング or thread + channel の簡易実装で可)
- 出力は呼び出し側で Operation Log に流せるよう全部返す

### 2. plan_pull / execute_pull(`src/git/ops.rs` 追記)

- `pub fn plan_pull(repo: &Repository) -> Result<OperationPlan, GitError>`
  - blocker: upstream なし / detached / unborn / conflict 状態 / **dirty(staged or unstaged)**(ADR-0009: stash を提案)
  - 表示: 現在の behind 数(ローカル情報)+ warning「fetch により増減する可能性があります」
  - 現時点の upstream tip との in-memory merge で conflict 予測(参考情報として warning/blocker 表示。
    fetch 後に再判定するので plan 段階は warning 止まりで可)
- `pub fn execute_pull(repo: &Repository, repo_path: &Path) -> Result<PullOutcome, GitError>`
  1. `run_git(repo_path, ["fetch", "<remote>"])`(remote 名は upstream から解決)。失敗 → Failed(stderr 込み)
  2. fetch 後の upstream tip を再解決。HEAD と同一 or HEAD が子孫 → `PullOutcome::UpToDate`
  3. **fast-forward 可能**(HEAD が upstream の祖先)→ branch ref を upstream tip へ + `checkout_tree` safe + `set_head` 相当の
     WT 同期(T013 の execute_checkout の安全手順を再利用)→ `PullOutcome::FastForward { to }`
  4. 分岐している → `repo.merge_commits(&head, &upstream, None)` の **in-memory index** で conflict 判定:
     - conflict → **何もせず** `GitError::Other("merge would conflict: <files>...")` で Failed
       (repo state を MERGING にしない。abort 不要)
     - clean → write_tree → merge commit 作成(parents = [head, upstream]、message = `Merge remote-tracking branch '<upstream>'`)
       → index 同期 + `checkout_head` safe(T016 execute_cherry_pick の手順を再利用)→ `PullOutcome::Merged { commit }`
  - 戻り値に fetch の stdout/stderr 要約を含めて Operation Log に流す
- preflight: 既存 preflight_check(HEAD 不変)

### 3. UI

- Pull ボタン → plan モーダル(behind / warnings / blockers)→ confirm → execute → oplog 記録(成功/失敗とも)→
  reload + footer(`pull: fast-forward to <short>` 等)。失敗時は Bottom Panel 自動オープン(既存機構)
- 実行は同期で可(ローカル file remote はミリ秒。実ネットワークで UI が固まるのは既知制限としてメモ —
  バックグラウンド化は別チケット)

### 4. headless env / テスト

- `KAGI_PULL=1` + `KAGI_AUTO_CONFIRM=1`
- `tests/pull_test.rs`(新規): fixture 同様の **bare remote 構成を tempdir に自作**して:
  - behind のみ → FastForward(WT 内容が更新される)
  - 分岐・conflict なし → Merged(merge commit の parents 2)
  - 分岐・conflict あり → Failed + **WT/index/HEAD 完全無傷 + repo state クリーン**を assert
  - up to date / dirty blocker / upstream なし blocker
  - fetch 失敗(remote ディレクトリを消す)→ Failed + 無傷

## 実装メモ

### 経緯
初代 agent が API エラーで死亡(バックエンドまで実装済み)→ 継続 agent も即ストール → **PM(Fable)が直接完成**させた。

### PM レビューで発見・修正した重大バグ(ref 順序問題)
**「ref を先に動かしてから checkout」すると safe checkout の baseline が新 tree になり差分ゼロ扱い → WT が古いまま残る**。
- FF パス: reference() → checkout_tree の順だった → checkout_tree(旧 baseline)→ reference() に修正
- merge パス: commit(Some("HEAD")) で ref が先に動いていた → commit(None) → checkout_tree → reference() に修正
- **cherry-pick(T016)にも同じ潜在バグがあり修正**(既存ファイルを変更する commit の pick で WT が stale になっていた)
- 回帰テスト: pull_test に「既存ファイル変更の FF / merge」、ops_test に「既存ファイル変更の cherry-pick」を追加

### 構成
- src/git/cli.rs: run_git(GIT_TERMINAL_PROMPT=0 / LC_ALL=C / 60s timeout / shell 非経由)。timeout 時に子プロセスが残るのは既知制限
- plan_pull: blocker = no-upstream / detached / unborn / conflict / dirty。behind はローカル知識の旨を title に明記
- execute_pull: fetch(CLI) → UpToDate / FF / in-memory merge(conflict なら無傷で Failed)
- UI: PullPlanModal + render_plan_modal_card(checkout モーダルと共通化)+ confirm_pull(preflight → execute → oplog → reload)
- env: KAGI_PULL=1(+KAGI_AUTO_CONFIRM=1)
