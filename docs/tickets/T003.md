# T003: working tree status を取得して表示する

- Status: in-progress
- 担当: Sonnet subagent
- 依存: T002

## 背景

「いま repo がどういう状態か」の常時可視化は本プロダクトの最重要価値(requirements M-02)。
その第一歩として working tree status を内部モデルに取り込み、画面に表示する。
status モデルは後続の plan/preflight(T013 以降の dirty 検査)でも使う基盤になる。

## スコープ

1. ドメインモデル(architecture.md §3)を `src/git/` に導入:
   - `WorkingTreeStatus { staged: Vec<FileStatus>, unstaged: Vec<FileStatus>, untracked: Vec<PathBuf>, conflicted: Vec<PathBuf> }`
   - `FileStatus { path: PathBuf, change: ChangeKind }`
   - `ChangeKind { Added, Modified, Deleted, Renamed { from: PathBuf }, TypeChange }`
   - `WorkingTreeStatus::is_dirty()` ヘルパー(後続 ticket の preflight 用)
2. backend に status 取得を実装:
   - `pub fn working_tree_status(repo: &Repository) -> Result<WorkingTreeStatus, GitError>` 相当
   - git2 の `statuses()` + `StatusOptions`(untracked を含める、untracked ディレクトリは再帰、ignored は除外)
   - index 側ビット(INDEX_*)→ staged、workdir 側ビット(WT_*)→ unstaged に振り分け。両方立つファイルは両方に出す
   - conflicted は `CONFLICTED` ビットで判定
   - rename 検出は staged(INDEX_RENAMED)のみで可。無理に highfidelity にしない
3. UI: repo open 成功時に status も取得し、ウィンドウに表示:
   - 「Staged (N)」「Unstaged (N)」「Untracked (N)」「Conflicted (N)」のグループ見出し + ファイルパス一覧(各グループ最大20件で truncate 表示可)
   - clean な repo では「Working tree clean」を表示
   - 取得は起動時1回の同期実行でよい(バックグラウンド化は後続 ticket)
4. **integration test を導入する**(本プロジェクト初のテスト):
   - `tests/status_test.rs`。tempdir に `std::process::Command` で `git` CLI を呼んで小さな repo を組み立て、staged / unstaged / untracked / clean の4ケースを assert
   - dev-dependency に `tempfile` を追加してよい
   - conflicted ケースのテストは任意(merge conflict の組み立てが必要。やらない場合はその旨メモに書く)
   - **テストが書き込むのは tempdir 配下のみ。それを assert すること**

### git2 API の注意点

- `StatusEntry::path()` は `Result<&str, Error>` を返す(Option ではない)
- `StatusEntry::path_bytes()` は `&[u8]` を返す(Option ではない)
- `WT_NEW` ビットが「untracked」。これは `unstaged` に入れず `untracked` に分類する
- `recurse_untracked_dirs(true)` を指定しないとディレクトリ内のファイルが個別に列挙されない
- `renames_head_to_index(true)` を指定しないと INDEX_RENAMED ビットが立たない

### conflicted テストを書かなかった理由

merge conflict 状態を programmatically に構築するには:
1. 同一行を変更する 2 つのブランチを作成
2. `git merge --no-commit` で conflict を残したままにする
3. git2 が `CONFLICTED` ビットを認識する状態にする

という手順が必要で、fixture 組み立てコードが大幅に複雑になる。MVP では `CONFLICTED` ビットの判定コードは実装済み (`status.rs` L116–L122)。後続チケットで必要になった時点でテストを追加する。
