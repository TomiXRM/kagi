# T004: commit log を取得して内部モデルに変換する

- Status: in-progress
- 担当: Sonnet subagent
- 依存: T002

## 背景

commit graph(本プロダクトの中心機能)の入力データとなる commit DAG を取得する。
T006 の graph layout はこの出力(topo order の `Vec<Commit>`)をそのまま入力に取るため、
**順序の保証(子が親より先)と parents の正確さ**が品質のすべて。

## スコープ

1. ドメインモデル(architecture.md §3)を `src/git/` に導入:
   - `CommitId(String)` newtype(40-hex。`short()` ヘルパー(先頭8桁)付き)
   - `Signature { name: String, email: String, time: i64 }`(epoch 秒。表示整形は UI 側)
   - `Commit { id: CommitId, parents: Vec<CommitId>, author: Signature, committer: Signature, summary: String, message: String }`
   - parents[0] が first parent であることを保証(doc comment に明記)
2. backend に log 取得を実装(例: `src/git/log.rs`):
   - `pub fn commit_log(repo: &Repository, limit: usize) -> Result<Vec<Commit>, GitError>`
   - revwalk で**全 ref から到達可能な commit**を列挙: `push_glob("refs/heads/*")` + `push_glob("refs/remotes/*")` + `push_glob("refs/tags/*")` + HEAD(detached 対応のため `push_head()` も。unborn は空 Vec を返す)
   - ソート: `Sort::TOPOLOGICAL | Sort::TIME`(新しい順、**子が必ず親より先**)
   - `limit` で打ち切り(呼び出し側 MVP 値: 10,000)
   - 非 UTF-8 の author / message は lossy 変換で受ける(panic しない)
3. UI: 表示は最小限でよい。「Commits: N」と先頭3件の `short id + summary` を既存の details 表示に追加(リスト UI は T008)
4. integration test(`tests/log_test.rs`):
   - tempdir + git CLI で「分岐 + merge を含む repo」を組み立て、以下を assert:
     - 全 commit が取得される(ref から到達可能なもの)
     - merge commit の parents が2つで、parents[0] が first parent(`git log --pretty=%P` と突き合わせ)
     - topo order: すべての commit について、その親はリスト内でより後ろに位置する
     - limit が効く
     - summary / message の取得(複数行 message で summary が1行目になる)
   - 既存 tests/status_test.rs のヘルパー(git() 等)は重複定義してよい(共通化リファクタは禁止。tests/common/ を作るのは可)

## リスク

- stash は `refs/stash` にいるため glob に含めなければ出ない(意図どおり。stash 表示は T015/T005 で別扱い)
- `Sort::TOPOLOGICAL` だけだと同世代の順序が不定 → `| Sort::TIME` で安定化。ただし git CLI の `--topo-order` と完全一致はしなくてよい(「子が親より先」だけが不変条件)
- annotated tag の peel: `push_glob("refs/tags/*")` は revwalk が自動で commit まで peel する(エラーになる場合は報告)
