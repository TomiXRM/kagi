## 背景

本プロダクトの中心機能。ADR-0003 で決定した「行単位 lane 割り当ての事前計算方式」を
**UI にも git2 にも依存しない pure Rust モジュール**として実装する。

### データ構造(architecture.md §3 準拠)

```rust
pub struct GraphLayout { pub rows: Vec<GraphRow>, pub lane_count: usize }

pub struct GraphRow {
    pub commit: CommitId,      // src/git の CommitId を再利用(git2 非依存の型なので可)
    pub lane: usize,           // この commit の node が乗る lane
    pub edges: Vec<GraphEdge>,
}

pub struct GraphEdge {
    pub from_lane: usize,      // 行の上端での lane
    pub to_lane: usize,        // 行の下端での lane
    pub kind: EdgeKind,
}

pub enum EdgeKind {
    Pass,       // この行を素通りする辺(from == to)
    IntoNode,   // 上から来てこの行の node に入る辺(to_lane == row.lane)
    OutOfNode,  // この行の node から親へ出ていく辺(from_lane == row.lane)
}
```

ref バッジ(HEAD / branch / tag)は **graph モジュールの責務外**(T008/T009 で UI 側が
`CommitId → refs` の map を作って行に重ねる)。色も lane index から UI 側で決める。

### アルゴリズム(この契約どおりに実装する)

入力: `&[Commit]`(topo order、子が親より先。T004 の出力)
状態: `active: Vec<Option<CommitId>>`(lane i が「次に待っている commit」)

各 commit C について(上から1行ずつ):

1. `waiting` = `active[i] == Some(C.id)` なる全 lane index
2. **node lane の決定**:
   - `waiting` が空(branch tip / 新規 root)→ 最左の `None` slot、なければ末尾に新 lane を確保
   - `waiting` が非空 → `lane = min(waiting)`
3. **edge 生成(行の上端 → 下端)**:
   - 各 `j ∈ waiting`: edge `(j → lane, IntoNode)` を追加し、`j != lane` の lane は解放(`active[j] = None`)
   - その他の lane k で `active[k] = Some(X)`(X ≠ C): edge `(k → k, Pass)`
4. **parents の配置(行の下端)**:
   - parents が空(root)→ `active[lane] = None`
   - parents[0](first parent)→ `active[lane] = Some(parents[0])`、edge `(lane → lane, OutOfNode)`
     - **例外**: parents[0] が既に別 lane j で待たれている場合は新たに占有せず `(lane → j, OutOfNode)` を追加し `active[lane] = None`(分岐合流の重複 lane を防ぐ)
   - parents[1..](merge 元)各 p:
     - p が既に lane j で待たれている → edge `(lane → j, OutOfNode)` のみ
     - 待たれていない → 最左の `None` slot(なければ末尾)に lane j を確保、`active[j] = Some(p)`、edge `(lane → j, OutOfNode)`
5. 末尾の連続する `None` lane は捨ててよい(`lane_count` は全行を通じた最大 lane 数 + 1)

公開 API: `pub fn layout(commits: &[Commit]) -> GraphLayout`

### 不変条件(テスト・debug_assert で守る)

- 各行の node はちょうど1つ。`row.lane` 上の IntoNode/OutOfNode の端点と一致する
- ある行の下端で lane j に出ている辺(to_lane == j の OutOfNode / Pass)は、次に lane j に関わる行まで連続している(辺が途切れない)
- 直線履歴では全行 lane 0、edges は Pass なし
- input が空なら rows 空・lane_count 0

### ステップ → 関数対応表

| T006 契約ステップ | 実装箇所 |
|---|---|
| Step 1: waiting lane 収集 | `layout()` 内 `waiting` 変数の生成ロジック |
| Step 2: node lane の決定 | `waiting.is_empty()` 分岐 + `find_or_push_free_lane()` |
| Step 3: top-half edge 生成 | IntoNode ループ + Pass ループ |
| Step 4: parents 配置・bottom-half edge | first-parent 例外分岐 + merge parent ループ |
| Step 4 例外: first-parent 既待ち | `find_lane_for()` で検出し OutOfNode(lane→j) + active[lane]=None |
| Step 5: lane_count 追跡 | `highest_used` 計算 + `max_lanes.max(...)` |

### 判断事項・注意点

- **Step 4 例外**: `find_lane_for(active, parents[0])` で既に別 lane j が待っている場合、node lane を解放して OutOfNode(node_lane → j) を生成。これを漏らすと同一 commit が 2 lane で待たれ lane が無限増殖する。
- **debug_assert 条件**: root tip（waiting も parents も空）は edges が空になるため、「directed edge が存在する場合のみ node_lane を含む」という条件に緩和した。
- **lane_count**: `active` の最大 is_some() インデックス + 1 と node_lane + 1 の大きい方を毎行更新することで、行を跨いだ最大値を追跡。
