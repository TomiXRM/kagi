# T013: checkout branch を plan 確認付きで実装する(OperationController 導入)

- Status: in-progress
- 担当: Sonnet subagent
- 依存: T005, T010

## 背景

本プロダクトの核心(requirements M-07 / M-11、ADR-0004)。**plan → confirm → preflight → execute → verify**
の操作パイプラインをここで導入し、T014〜T016 の全操作がこのレールに乗る。
checkout は Guarded 操作: 「working tree の変更を失う可能性があれば実行しない」。

## スコープ

### 1. ドメイン(lib 側、`src/git/ops.rs` 新規)

```rust
pub struct StateSummary { pub head: String, pub dirty: String }  // 例: "branch: main", "1 modified, 1 untracked"
pub struct OperationPlan {
    pub title: String,            // "Checkout branch 'feature/two'"
    pub current: StateSummary,
    pub predicted: StateSummary,
    pub warnings: Vec<String>,    // 実行は可能だが注意(例: untracked あり)
    pub blockers: Vec<String>,    // 1つでもあれば実行不可(UI は Execute を無効化)
    pub recovery: String,         // 失敗時の復旧手順テキスト
}
```

- `pub fn plan_checkout(repo: &Repository, branch: &str) -> Result<OperationPlan, GitError>`
  - blocker 条件(ADR-0004 の Guarded ポリシー):
    - 対象 branch が存在しない / 既に HEAD / repo が conflict 状態
    - **staged または unstaged の変更がある**(untracked は warning 止まり。
      「変更を失う可能性があるため拒否。stash してから再実行してください」と blocker 文言に含める)
  - predicted: `head: "branch: <target>"`, dirty は現状維持
  - recovery: `git checkout <元branch>` で戻れる旨 + reflog の案内
- `pub fn execute_checkout(repo: &Repository, branch: &str) -> Result<(), GitError>`
  - `repo.checkout_tree(&obj, Some(CheckoutBuilder::new().safe()))` → `repo.set_head("refs/heads/<branch>")`
  - **safe モード必須**(force を絶対に使わない)
- preflight / verify はヘルパとして ops.rs に:
  - `pub fn preflight_check(repo, plan_head: &Head) -> Result<(), GitError>`: 現 HEAD が plan 時と同じか(違えば中断エラー)
  - verify は execute 後に snapshot を取り直し、HEAD が target になっていることを確認(UI 層で実施)

### 2. UI

- **Sidebar(左、幅 200px)を新設**: `LOCAL BRANCHES` 一覧(HEAD に ✓)。クリックで checkout plan モーダルを開く
  (HEAD branch クリックは何もしない)。Sidebar は architecture.md §2 の最小版。remote/tags/stashes は後続
- **Plan モーダル**(画面中央のオーバーレイ): title / Current → Predicted / warnings(黄)/ blockers(赤)/
  recovery / ボタン `[Cancel]` `[Checkout]`。**blocker があれば Checkout ボタンを出さない**。
  Cancel で閉じる(デフォルト動作が安全側 = 何も起きない)
- 実行フロー(`KagiApp` のメソッド): confirm クリック → preflight → execute → **snapshot 再取得 → rows/details/badges/header 全再構築** → 成功ログ
  - 再構築のため `KagiApp::reload_from(repo_path)` 的なメソッドを作る(from_snapshot のロジックを再利用)
  - 失敗時: モーダルにエラー + recovery を表示(クラッシュしない)
- ログ([kagi] 形式): `plan: checkout <branch> blockers=N warnings=N` / `executed: checkout <branch>` / `verified: HEAD=<branch>`

### 3. headless 検証用 env(fixture 専用。README 等には書かない)

- `KAGI_PLAN_CHECKOUT=<branch>`: 起動後に plan を生成しモーダルを開き、plan ログを出す
- `KAGI_AUTO_CONFIRM=1`: 上と併用で、blocker がなければ自動で confirm まで進める(テスト専用と doc comment に明記)

### 4. integration test(`tests/ops_test.rs` 新規)

- clean repo: plan に blocker なし、predicted が target、execute で HEAD 移動 + working tree 更新
- dirty repo(unstaged modified): plan に blocker あり
- untracked のみ: blocker なし・warning あり、execute 後も untracked が残る
- 存在しない branch / 既に HEAD: blocker
- preflight: plan 後に repo 側で branch を進めて(別 commit)preflight が中断エラーを返す
- **全テスト tempdir のみ。execute 系テストは必ず自前で作った repo に対して行う**

## リスク

- `checkout_tree` の safe モードでも untracked との衝突で失敗しうる → 失敗を GitError で返しモーダル表示(クラッシュ・force 昇格をしない)
- set_head と checkout_tree の順序: checkout_tree(target tree)→ set_head が正しい(逆だと workdir が古いまま)
- モーダルのオーバーレイ: gpui では絶対配置(`.absolute().inset_0()` 等)の div で重ねられる。registry の examples を確認
- KAGI_AUTO_CONFIRM はテスト専用であることを doc comment で明示(誤って常用されない名前にしている)
