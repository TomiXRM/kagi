# T015: stash push / list / apply を実装する

- Status: in-progress
- 担当: Sonnet subagent
- 依存: T013

## 背景

requirements M-09。checkout の blocker(dirty)からの導線として最重要:
「stash してから checkout」がアプリ内で完結するようになる。
- stash push: Guarded(working tree を変更するが、内容は stash に保存される)
- stash apply: Guarded(conflict の可能性。**pop ではなく apply** = stash は消えないので安全側)
- stash list: 読み取り(T005 で取得済み。UI に出すだけ)

## スコープ

### 1. ドメイン(`src/git/ops.rs` 追記)

- `pub fn plan_stash_push(repo: &mut Repository, message: Option<&str>) -> Result<OperationPlan, GitError>`
  - blocker: 変更が何もない(staged/unstaged/untracked すべて空)/ conflict 状態
  - warning: untracked も含めて stash する旨(`-u` 相当。include_untracked を使う)
  - predicted: working tree clean / stash が1件増える
  - recovery: `git stash pop` / `git stash list` の案内
- `pub fn execute_stash_push(repo: &mut Repository, message: Option<&str>) -> Result<(), GitError>`
  - `repo.stash_save2(&sig, message, Some(StashFlags::INCLUDE_UNTRACKED))`(API 名は docs.rs/git2/0.21 で確認)
  - signature は repo の config から(なければ "kagi <kagi@local>" フォールバック)
- `pub fn plan_stash_apply(repo: &mut Repository, index: usize) -> Result<OperationPlan, GitError>`
  - blocker: index が範囲外 / conflict 状態 / **working tree が dirty**(apply の conflict で
    既存変更と混ざる事故を防ぐ。clean でのみ apply 可、が MVP ポリシー)
  - predicted: stash の内容が working tree に戻る(stash 自体は残る = apply)
  - recovery: conflict したら `git checkout -- .` ではなく(危険)、`git stash list` で stash が残っている旨 +
    conflict ファイルの手動解決を案内
- `pub fn execute_stash_apply(repo: &mut Repository, index: usize) -> Result<(), GitError>`
  - `repo.stash_apply(index, None)`。**stash_pop / stash_drop は実装しない**(drop は Destructive、v0.2)
- preflight: 既存 `preflight_check` に加え、stash 操作では **stash 件数が plan 時と同じ**ことも確認する
  ヘルパを足してよい(`preflight_check_stash(repo, plan, expected_stash_count)` 等)

### 2. UI

- Sidebar に `STASHES` セクション追加(T005 のデータ。`stash@{N} <message>`)
- Header の status 部分の隣に `Stash` ボタン(dirty のときだけ表示)→ stash push plan モーダル
  (message 入力は T014 の簡易テキスト入力を再利用。空なら None)
- Sidebar の stash 行クリック → stash apply plan モーダル
- どちらも T013/T014 と同じモーダル UI(Current → Predicted / warnings / blockers / recovery / Cancel・実行)
- 実行後 reload(stash 件数・status・必要なら graph が更新される)
- ログ: `plan: stash-push|stash-apply ...` / `executed: ...` / `verified: ...`(stash 件数と status で検証)

### 3. headless 検証 env

- `KAGI_STASH_PUSH=1`(+ `KAGI_AUTO_CONFIRM=1`)
- `KAGI_STASH_APPLY=<index>`(+ `KAGI_AUTO_CONFIRM=1`)
- 非 auto-confirm 時はモーダル状態をセットする(T014 の PM 修正と同じパターン)

### 4. integration test(`tests/ops_test.rs` 追記。既存テスト変更禁止)

- push 正常系: dirty repo → push 後 clean + stash 1件増
- push blocker: clean repo
- apply 正常系: clean repo + stash あり → apply 後 dirty(内容一致)+ **stash が残っている**(apply であること)
- apply blocker: dirty repo / index 範囲外
- push → apply の round-trip でファイル内容が戻る
