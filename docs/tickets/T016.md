# T016: cherry-pick dry-run preview を実装する

- Status: in-progress
- 担当: Sonnet subagent
- 依存: T013

## 背景

requirements M-10 / ADR-0002・0005。**本プロダクトの目玉機能**:
libgit2 の in-memory merge で「実行したら何が起きるか(変更ファイル / conflict 有無)」を
**working tree に一切触れずに**予測し、conflict が予測される cherry-pick は実行させない。

## スコープ

### 1. ドメイン(`src/git/ops.rs` 追記)

- `OperationPlan` に `pub preview_files: Vec<FileStatus>` を追加(他操作では空。modal で「この操作で変わるファイル」を表示)
- `pub fn plan_cherry_pick(repo: &Repository, id: &CommitId) -> Result<OperationPlan, GitError>`
  - **dry-run preview(核心)**: `repo.cherrypick_commit(&commit, &head_commit, 0, None)` で
    **in-memory index** を作る(working tree 非接触。API は docs.rs/git2/0.21 で確認)
    - `index.has_conflicts()` → blocker「conflict が予測されるため実行できません」+ conflict ファイル一覧
      (`index.conflicts()` から path を列挙)
    - conflict なし → in-memory index から tree を一時的に書き(`index.write_tree_to(repo)`)、
      HEAD tree との diff で **preview_files** を作る(T011 の delta→ChangeKind 変換を再利用してよい)
  - その他の blocker:
    - working tree が dirty(staged / unstaged。untracked は warning)
    - conflict 状態の repo
    - 対象が merge commit(parents > 1。mainline 選択は scope 外)
    - 対象 commit が HEAD と同一
    - HEAD が unborn / detached(MVP は attached branch 上のみ)
    - preview の結果、変更が空(= 既に適用済み)→ blocker「適用しても変更がありません」
  - predicted: HEAD branch に新 commit が1つ載る(`<short> '<summary>' を <branch> に適用`)
  - recovery: cherry-pick 後に戻したい場合の案内(`git reflog` + 直前 commit の sha を控える旨)。
    **reset コマンドの提示はしない**(`git revert <new-sha>` を案内する)
- `pub fn execute_cherry_pick(repo: &Repository, id: &CommitId) -> Result<CommitId, GitError>`
  - plan と同じ `cherrypick_commit` で in-memory index を再生成(**working tree 方式の `repo.cherrypick()` は使わない**。
    repo state を CHERRYPICK にしない = abort 不要の安全設計)
  - conflict があればエラーで中断(preflight 相当の二重チェック)
  - `index.write_tree_to(repo)` → `repo.commit(Some("HEAD"), &original_author, &committer, &original_message, &tree, &[&head_commit])`
    - author / message は元 commit から引き継ぐ。committer は repo config(なければ fallback)
  - 戻り値は新 commit の id。**working tree と index には一切触れない**ため、
    実行後の checkout: `repo.checkout_head(Some(CheckoutBuilder::new().safe()))` で
    working tree を新 HEAD に同期する(clean 前提なので safe で必ず成功するはず。失敗したらエラー表示)
- preflight: 既存 `preflight_check`(HEAD 不変)を再利用

### 2. UI

- Detail Panel(commit 選択時)に `Cherry-pick onto <HEAD branch>` ボタン → plan モーダル
- モーダル: 既存スタイル + **preview_files セクション**(T018 のファイルツリー表示を再利用)
  - conflict 予測時: blocker 表示 + conflict ファイル一覧(赤)
- 実行後 reload(graph に新 commit が出る)
- ログ: `plan: cherry-pick <short> blockers=N preview_files=M` / `executed: cherry-pick <short> -> <new-short>` / `verified: ...`

### 3. headless 検証 env

- `KAGI_CHERRY_PICK=<sha>`(short 可)+ `KAGI_AUTO_CONFIRM=1`。非 auto-confirm はモーダルセット(既存パターン)

### 4. integration test(`tests/ops_test.rs` 追記。既存テスト変更禁止)

- 正常系: 別 branch の commit を clean repo に cherry-pick → 新 commit が HEAD に載る(message/author 一致、
  parent が旧 HEAD)、working tree にファイルが反映、status clean
- **conflict 予測**: 同一行を変更した2 branch で plan → blocker + conflict ファイル名 + working tree が無傷
- dirty WT → blocker / merge commit → blocker / HEAD 同一 → blocker / 適用済み(空)→ blocker
- preview_files: 正常系 plan の preview_files が実際の変更ファイルと一致
- **すべてのケースで「plan しただけでは repo が一切変わらない」を assert**(status / HEAD / branch tips)

## リスク

- `cherrypick_commit` の mainline 引数: 非 merge commit では 0 を渡す(docs 確認)
- in-memory index の `write_tree_to` は conflict があると失敗する → 先に has_conflicts() で弾く
- 実行後の working tree 同期(checkout_head safe)を忘れると「commit はあるのに WT が古い」状態になる → テストで WT 内容を assert
- OperationPlan へのフィールド追加で既存 ops テストのコンストラクト箇所がコンパイルエラーになる場合、
  既存テストの変更は禁止なので Default や builder ではなく**既存 plan_* 関数経由**で作られていることを確認(直接 struct を組み立てているテストがあれば最小修正可、ただし報告)
