# T017: error handling と operation log を整える(MVP 最終チケット)

- Status: in-progress
- 担当: Sonnet subagent
- 依存: T013–T016

## 背景

requirements M-12「操作前後の repo 状態をログに残し、失敗時に復旧手順を表示する」の完成 +
積み残し負債の解消で MVP を締める。

## スコープ

### 1. operation log(lib 側、`src/git/oplog.rs` 新規)

- 追記専用の JSON Lines ファイルに全操作を記録する:
  ```rust
  pub struct OpLogEntry {
      pub timestamp: i64,            // epoch 秒
      pub op: String,                // "checkout" / "create-branch" / "stash-push" / "stash-apply" / "cherry-pick"
      pub repo: String,              // workdir パス
      pub before: StateSummary,      // plan 時の current
      pub outcome: OpOutcome,        // Success { after: StateSummary } | Failed { error: String } | Refused { blockers: Vec<String> }
  }
  pub fn append_oplog(entry: &OpLogEntry) -> Result<PathBuf, GitError>  // 書いた先のパスを返す
  ```
- 保存先: `$HOME/.kagi/operations.jsonl`(`KAGI_LOG_DIR` env があればそちら優先 — テスト用)。
  ディレクトリは自動作成。書き込み失敗はアプリを止めない(stderr 警告のみ)
- JSON 整形は手書きで可(serde 追加禁止。フィールドは全部文字列/数値なのでエスケープだけ注意: `"` `\` 改行)
- unit test: tempdir(KAGI_LOG_DIR)に2件追記 → 2行になり各行が期待のフィールドを含む / 特殊文字エスケープ

### 2. UI 統合(`src/ui/mod.rs`)

- 全 confirm_*(checkout / create-branch / stash-push / stash-apply / cherry-pick)で
  実行結果(成功 / 失敗 / blocker 拒否)を `append_oplog` に記録
- **status footer**(ウィンドウ最下部、高さ22px)を新設:
  直近の操作結果を1行表示(成功=緑 / 失敗=赤 / それ以外は repo 名等の既定文言)。
  これで「操作の結果がどこに残るか」が画面上でも分かる
- 失敗時モーダルに recovery が表示されることを再確認(既存実装の確認のみ)

### 3. 負債解消(bin/lib の二重コンパイル)

- `src/main.rs` の `mod git; mod graph;` をやめ、**lib crate を使う**(`use kagi::git;` / `use kagi::graph;`)。
  `src/ui/` 内の `crate::git` / `crate::graph` 参照は `kagi::git` / `kagi::graph` に置換
  (`mod ui;` は bin 側に残す。lib に ui を入れない方針は不変)
- これで bin 側の dead_code 警告(`ChangeKind::label` / `old_lineno` 等)が消えるはず。
  **`cargo build` の警告 0 件**を完了条件とする(gpui 依存の future-incompat 警告は除く)

### 4. 仕上げ確認

- `cargo test` 全パス + fixture で全 headless 検証(checkout / create-branch / stash-push / stash-apply / cherry-pick)を一巡し、
  各操作後に `~/.kagi`(KAGI_LOG_DIR 指定先)へログが追記されることを確認
