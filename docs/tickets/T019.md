# T019: テキストオーバーフロー修正(ユーザー報告バグ)

- Status: in-progress
- 担当: Sonnet subagent
- 依存: T010, T012
- 発端: ユーザー報告「コミットを選択して右ペインが出ると、幅が変わって commit message と commit hash がオーバーフローする」

## 背景

gpui のテキスト折り返しは**単語境界でのみ**行われるため、切れ目のないトークンが container 幅を突き破る:
- 40桁の commit hash(Detail Panel の SHA フィールド)
- スペースなしの長い文字列(日本語長文、長い英数字トークン、email、長パス、長 branch 名)

再現 repo: 長い日本語1行 message + 長 ASCII トークン + 長 branch 名 + 深いパスで確認済み(PM 撮影のスクリーンショットあり)。

## スコープ

### 1. 調査(最初にやる)

`~/.cargo/registry/src/*/gpui-0.2.2/src/styled.rs` 等で利用可能なテキスト制御 API を確認する:
`truncate` / `text_ellipsis` / `line_clamp` / `whitespace_nowrap` / `overflow_hidden` / word-break 系の有無。
**break-all 相当がない場合**は後述の「ZWSP 挿入」方式を使う。

### 2. 共通ヘルパー(`src/ui/` の適切な場所)

```rust
/// スペースを含まない長い run に U+200B (zero-width space) を挿入して
/// 任意位置での折り返しを可能にする。表示幅には影響しない。
pub fn soft_wrap(text: &str, every: usize) -> String
```
- 連続 non-whitespace が `every`(例: 16)文字を超えたら ZWSP を挿入(chars() ベース、バイト境界禁止)
- unit test: ASCII 長トークン / 日本語長文 / 既にスペースがある文 / 空文字

### 3. 修正箇所(優先度順)

| 箇所 | 修正 |
|------|------|
| **Detail Panel: full SHA** | `soft_wrap(sha, 8)` で折り返し可能に(8桁ごと = short id 単位で読みやすい)。コンテナに `overflow_hidden` |
| **Detail Panel: Message** | 表示文字列を `soft_wrap(.., 16)` で構築 + コンテナ `overflow_hidden`。行数が多い場合に备え `line_clamp` 相当があれば 20 行程度に制限(なければ chars ベースで事前カット + 「…」) |
| **Detail Panel: Author / Committer 行**(email が長い) | 同上 soft_wrap |
| Detail Panel / diff パネル全体 | ルート container に `overflow_hidden`(何があってもパネル外に漏れない最後の砦) |
| commit list の ref バッジ(長 branch 名) | ラベルを 24 chars で truncate + 「…」(chars ベース) |
| ヘッダ行(長 branch 名) | `overflow_hidden` |
| Plan モーダルの blocker / recovery 長文 | `soft_wrap` 適用(モーダル幅は固定なので) |
| ファイルツリーの長いファイル名 / 圧縮ディレクトリ名 | 行に `overflow_hidden`(truncate は不要、はみ出さなければよい) |

## リスク

- ZWSP は等幅フォントでも幅0で描画されるはずだが、万一豆腐(□)が出るフォントなら別の方法(事前ハード改行)に切り替える → 実機確認必須
- soft_wrap をコピー機能(将来)に流用すると ZWSP が混入する → SHA 等の生値は別フィールドに保持し、表示用文字列にのみ適用する(doc comment に明記)
- `select()` 時の tree ログ等、表示文字列を検証に使っている箇所に ZWSP が入るとログ比較が壊れる → ログには生値を使う

## 実装メモ

### gpui text 制御 API 調査結果

`~/.cargo/registry/src/.../gpui-0.2.2/src/styled.rs` で確認した利用可能な API:

| API | 有無 | 備考 |
|-----|------|------|
| `overflow_hidden()` | ✓ | マクロ生成 |
| `whitespace_nowrap()` | ✓ | WhiteSpace::Nowrap |
| `text_ellipsis()` | ✓ | TextOverflow::Truncate(ELLIPSIS) |
| `truncate()` | ✓ | overflow_hidden + nowrap + ellipsis の組み合わせ |
| `line_clamp(n)` | ✓ | text_style.line_clamp = Some(n) |
| `word-break: break-all` 相当 | **なし** | CSS の word-break には未対応 → ZWSP 方式を採用 |

結論: break-all が存在しないため、連続 non-whitespace トークンの任意位置折り返しは ZWSP 挿入方式で実装。

### 未解決リスク

1. **ZWSP の実機描画確認未実施**: PM によるスクリーンショット確認が必要。万一フォントが ZWSP を豆腐(□)として描画する場合は、soft_wrap をハード改行 `\n` 方式に差し替える必要がある。
2. **ZWSP が mixed に**: `full_message` や `author_line` などの表示フィールドは ZWSP を含む。将来のコピー機能実装時は生値（commit message の raw 文字列など）を別途保持する設計が必要。`full_sha` (生値) と `sha_display` (ZWSP 入り) の分離はすでに実装済み。
3. **ログ/検証の raw 値使用**: `select()` 内のログは `detail.full_sha.as_ref().get(..8)` (生値) を使用しており、ZWSP は混入しない。
