# T025: Commit Panel UI(GitKraken 風の作業台)

- Status: in-progress
- 担当: Sonnet subagent
- 依存: T024, T018, T022
- 発端: ユーザーゴール(全要件は下記チェックリスト)

## 背景

「何を履歴に残すかを決める作業台」。diff 確認と stage 判断を同じ文脈で行えることが核心。
ゴール要件: unstaged/staged 一覧(件数付き)、flat/tree 切替、ファイル選択で diff、
file 単位 stage/unstage、status 表示(A/M/D/R/conflicted)、message 入力、
staged なしで commit ボタン無効、unstaged 残り警告、diff と staging 操作の同居。

## スコープ

### 1. WIP 行(エントリポイント)

- working tree が dirty のとき、commit list の**先頭に WIP 行**を表示
  (バッジ列に `WIP`、メッセージ部に `// WIP — N changes` 風。graph lane は HEAD 行と同 lane の上に置くだけでよい。
  graph layout 本体は変更しない)
- WIP 行クリック → 右ペインが **Commit Panel** になる(commit 選択とは排他)

### 2. Commit Panel レイアウト(上から)

```
┌─ Commit Panel ──────────────┐
│ Unstaged (N)      [flat|tree]│  ← 表示切替トグル(両セクション共通)
│   [M] src/a.rs        [Stage]│  ← 行 hover か常時で Stage ボタン
│   ...                        │
│ Staged (M)                   │
│   [A] src/b.rs      [Unstage]│
│ ─ diff viewer ───────────────│  ← 選択ファイルの diff(staged 側なら index diff)
│   (uniform_list, flex_1)     │
│ ─ commit ────────────────────│
│ [message 入力(1行 summary)]  │
│ ⚠ N unstaged change(s) not   │
│   included                   │
│        [Commit (M files)]    │  ← staged 0 or message 空で disabled 表示
└──────────────────────────────┘
```

- ファイル行: status 1字バッジ(A=green/M=yellow/D=red/R=blue/**C=conflicted は赤背景で操作不可**)+ パス + Stage/Unstage ボタン
- flat / tree 切替: `KagiApp` に `commit_panel_tree_view: bool`。tree は T018 の `build_file_tree` を再利用
  (flat は従来のパス一覧)。トグルは Unstaged/Staged 両方に効く
- ファイルクリック → 下部 diff(unstaged 行は `unstaged_file_diff`、staged 行は `staged_file_diff`)。
  キャッシュは「stage/unstage のたびに無効化」(状態が変わるため)
- stage/unstage 実行後: status を再取得してセクションを更新(`cx.notify`)。
  **plan モーダルは挟まない**(index のみの Safe 操作。即時反映が作業台としての要)。oplog 記録も不要
- message 入力: **gpui-component 0.5.1 の Input を導入して使う**(ADR-0006。`Cargo.toml` に
  `gpui-component = "0.5.1"` を追加 — このチケットでのみ Cargo.toml 変更を許可)。
  これにより **日本語 IME 入力対応**。API は docs.rs/gpui-component/0.5.1 と 0.5.1 タグの examples が一次資料
  (公式サイトは git main 準拠なので見ない)。Input の組み込みが難航した場合は T014 の簡易入力に
  フォールバックし、理由を報告(その場合 IME 非対応が既知制限)
- Commit ボタン: staged 0 件 or message 空なら無効(押せない見た目 + クリック無視)。
  押すと **plan_commit のモーダル**(T013 と同じパイプライン: unstaged 残り warning / preview_files = staged)
  → confirm で execute_commit → oplog 記録 → snapshot 再取得 + reload(WIP 行と graph が更新される)

### 3. headless 検証 env

- `KAGI_COMMIT_PANEL=1`: 起動時に WIP 行選択相当で Commit Panel を開き、
  `[kagi] commit-panel: unstaged=N staged=M` をログ
- `KAGI_STAGE_FILE=<path>` / `KAGI_UNSTAGE_FILE=<path>`: 起動時に1ファイル stage/unstage して件数ログ
- `KAGI_COMMIT_MSG=<msg>` + `KAGI_AUTO_CONFIRM=1`: plan_commit → execute まで(既存パターン)

### gpui-component Input の組み込み結果

**フォールバック採用 (T014 パターン)** — 理由:

`InputState::new` は `&mut Window` を必要とするが、`KagiApp` の状態初期化は `from_snapshot()` (Window コンテキスト外) で行われており、`KagiApp` の方法 (`open_commit_panel` 等) は `&mut Context<Self>` しか持たない。`Entity<InputState>` を `render()` 内で遅延生成する手法は `window.use_keyed_state` (RenderOnce trait 用) で、`Entity<KagiApp>` の Render 実装とは設計が合わない。この制約を回避するには KagiApp の構造を大幅リファクタリングする必要があり、T025 スコープ外と判断。

結果: `gpui_component::init(cx)` は `run_app` に追加済み。将来の gpui-component 他コンポーネント (Scrollbar, Tree など) の導入基盤は整備済み。commit message 入力は T014 パターン (KeyDownEvent ハンドラ) で実装 — IME 非対応が既知制限。

### 設計メモ

- WIP 行は `uniform_list` の上に `flex_col` で独立した div として描画。index offset 問題を回避。
- commit panel open と commit 選択は排他: `select()` 呼び出しで `commit_panel_open = false` に。
- diff キャッシュ無効化: stage/unstage 後は `panel.reload_status()` を呼び、`diff_view = None` / `selected_file = None` にリセット。
- flat/tree 切替: `CommitPanelState::tree_view` フラグで両セクション共通。
- plan_commit → execute_commit パイプライン: T013 / T015 パターンと同一構造。oplog 記録あり。
