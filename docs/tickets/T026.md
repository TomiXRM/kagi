# T026: commit message 入力を gpui-component Input に置き換え(IME 対応)

- Status: in-progress
- 担当: Sonnet subagent
- 依存: T025, ADR-0006

## 背景

T025 で gpui-component 0.5.1 を導入したが、Input(`InputState`)は `InputState::new(&mut Window, ..)` が
必要で、`KagiApp::from_snapshot`(Window 外)での初期化と噛み合わずフォールバックした。
現状の commit message 入力は ASCII のみ(**日本語 IME 不可**)。

## スコープ

- `InputState` の生成を **Window が手に入る場所**に移す:
  - 案A: WIP 行クリックの `cx.listener`(`&mut Window` を受け取れる)内で `cx.new(|cx| InputState::new(window, cx))` を生成して `KagiApp.commit_input: Option<Entity<InputState>>` に保持
  - 案B: `open_window` のクロージャ内(Window あり)で起動時に一度だけ生成
  - 0.5.1 タグの examples(input)で正確な生成・読み出し API を確認(docs.rs/gpui-component/0.5.1)
- commit message の読み出し(plan_commit へ渡す値)を InputState から取得
- 見た目を既存パネルの配色に合わせる(可能な範囲で)
- KAGI_COMMIT_MSG env パスは現状維持(InputState を経由しない直接 plan でよい)
- フォーカス管理: パネルを開いたら入力にフォーカス(できれば)

## 実装メモ

### 実装日: 2026-06-12

### 変更ファイル
- `src/ui/mod.rs` — KagiApp に `commit_input: Option<Entity<InputState>>` フィールドを追加、`open_commit_panel` に `&mut Window` 引数を追加、`render_commit_panel` で `Input::new` を使用、`open_commit_plan_modal`/`confirm_commit` に `cx` 引数を追加してInputStateから値を読み出す
- `src/ui/commit_panel.rs` — `can_commit` に `#[allow(dead_code)]` を追加（T026 では render 側で判定するため）

### 採用した案

**案A採用**: WIPクリックの `cx.listener` 内で `open_commit_panel(window, cx)` を呼び、その中で `cx.new(|cx| InputState::new(window, cx))` を生成。`KagiApp.commit_input: Option<Entity<InputState>>` に保持。

### 設計メモ

- `InputState` 生成は `open_commit_panel(&mut self, window: &mut Window, cx: &mut Context<Self>)` 内で lazy に実施。`commit_input` が None のとき一度だけ生成(パネル再オープン時は既存の InputState を再使用し入力内容が保持される)
- `commit_panel.commit_msg`(String) はヘッドレス KAGI_COMMIT_MSG パスで引き続き使用。UI 経由では InputState.value() が優先される
- `can_commit` ロジックを `render_commit_panel` で InputState の値から計算するよう変更
- フォーカス: パネルを開いた直後に `input_entity.update(cx, |state, cx| state.focus(window, cx))` を呼んでいるが、テーマ初期化タイミングの都合で実機確認が必要
- `handle_commit_msg_key` は T025 の旧 KeyDownEvent ベースの実装。T026 以降は InputState が処理するため呼ばれないが、`#[allow(dead_code)]` を付けて残存

### 未解決リスク

- InputState が ActiveTheme を要求するため、`gpui_component::init(cx)` が呼ばれていない環境ではパニックが起きる可能性があるが、`run_app` 内で確実に呼ばれているので問題なし
- パネルを開いた直後のフォーカスは Window が存在する環境でのみ動作する。headless 起動では commit_input は None のままなのでフォーカス処理はスキップされる
- 実機での日本語 IME 入力確認は PM が行う
