# T029: 外部変更の自動追従(.git 監視によるイベント駆動リフレッシュ)

- Status: in-progress
- 担当: Sonnet subagent
- 依存: T005

## 背景

ユーザー報告「kagi を開いた後にターミナルで branch を変更しても追従されず、再起動したら追従した」。
GitKraken 等は FS イベントで `.git` を監視して追従する。同方式を導入する。
(architecture.md §6 で v0.2 送りにしていた file watcher の前倒し)

## スコープ

1. **依存追加(このチケットのみ許可)**: `notify = "6"`(クロスプラットフォーム FS watcher。
   最新バージョンと推奨 API は docs.rs で確認。`notify-debouncer-mini` を足してもよい)
2. watcher(`src/ui/` か bin 側の新モジュール。lib には入れない):
   - 監視対象: `<repo>/.git` ディレクトリ(再帰)。ただしイベントは
     `HEAD` / `refs/` / `packed-refs` / `index` / `MERGE_HEAD` / `CHERRY_PICK_HEAD` に
     関係するパスだけ採用(objects/ の大量イベントは無視)
   - **デバウンス 500ms**(git 操作は短時間に多数のイベントを出す)
3. gpui への橋渡し:
   - watcher は別スレッド → channel へ。UI 側は gpui の async(`cx.spawn` / `background_executor` +
     `WeakEntity<KagiApp>` 更新)で受けて `reload()` を呼ぶ
   - **gpui 0.2.2 の正しい async パターンを registry の examples(animation.rs 等)と
     docs.rs で確認してから書く**(想像で書かない)。channel は std mpsc + ポーリングでも
     smol/async チャネルでもよいが、busy-loop にしない
4. reload 時の挙動:
   - snapshot 再取得 → rows/details/header 再構築(既存 `reload()`)
   - 可能なら選択中 commit を CommitId で追跡して選択維持(消えていたら解除)。困難なら解除でよい(メモに書く)
   - commit panel が開いていたら staging 状態も再読込
   - footer + ログ: `[kagi] refreshed (external change)`
5. 自分自身の操作(checkout 等)で二重 reload になるのは許容(デバウンスで実害なし)
